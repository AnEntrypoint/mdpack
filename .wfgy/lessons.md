## 2026-07-14 -- gm-plugkit crash-loop from duplicate supervisor trees racing over one spool dir
Goal (G): get the gm spool serving in C:\dev\pxpack so the actual task (pxpipe-equivalent for arbitrary .md files) could start.
What drifted / what went wrong: first `bun x gm-plugkit@latest spool` on a non-git directory produced a wasm panic (`gm_dir: project root resolution failed` via `git rev-parse`) on the first `instruction` dispatch. After `git init -b main`, a second bare retry left the *original* crashed process still alive underneath, and a fresh `spool` bootstrap spawned a second independent supervisor/watcher tree pointed at the same `.gm/exec-spool` dir. The two supervisors fought over `.status.json`/`.watcher.lock`, each restarting the watcher every ~10s with reason `planned-restart-after-heartbeat-stale`, so the watcher never got past boot far enough to print its own startup banner or process any queued verb -- a true crash loop, not just a slow cold start.
Fix / resolution: `Get-CimInstance Win32_Process -Filter "Name='bun.exe'"` with `CommandLine` revealed two full supervisor->wrapper->watcher process trees for gm-plugkit. Killed every pid in both trees, deleted stale `.status.json`/`.watcher.lock`/`.supervisor.json`/`.supervisor.pid`/`.boot-active.json` plus stale `out/instruction-*.json` and queued `in/instruction/*.txt`, then ran `spool` exactly once more. Single supervisor came up clean, watcher reached "spool watcher running", and the next `instruction` dispatch answered in under 2s.
Generalizes to: when a gm-plugkit watcher crash-loops with `planned-restart-after-heartbeat-stale` and no boot banner ever appears in `.watcher.log`, check for duplicate supervisor processes (`Get-CimInstance Win32_Process -Filter "Name='bun.exe'"`, filter CommandLine for `gm-plugkit`/`plugkit-wasm-wrapper`) before assuming it's an environment/wasm defect -- killing every stray process in every stray tree and clearing stale spool state files is the actual fix, a bare re-run of `spool` while an old crashed process is still alive just adds a second racing tree instead of replacing the first.

## 2026-07-14 -- GitHub Actions self-trigger guard using contains() false-positives on its own description
Goal (G): add a release workflow to AnEntrypoint/mdpix that bumps the npm version and publishes on every push to main, without the bot's own bump-commit push re-triggering itself in an infinite loop.
What drifted / what went wrong: the loop guard was written as `if: "!contains(github.event.head_commit.message, 'chore(release):')"`. The very first real commit that added this workflow described the guard in its own commit message body ("...via a chore(release): message check"), which itself contains the literal substring the guard checks for -- so the workflow's own introducing commit was skipped by its own guard. This was only caught by actually watching the live run (`gh run list` showed `skipped`), not by reading the YAML.
Fix / resolution: changed `contains()` to `startsWith()`. `npm version patch -m "chore(release): %s"` always places that exact prefix at the start of the message, so `startsWith` matches only real bump commits and ignores any commit that merely mentions the phrase in prose. Verified with a plain `node -e` string check, then re-pushed and watched a second live run go green (`gh run watch --exit-status`), confirmed `npm view mdpix version` and the `v0.1.1` git tag landed, and confirmed no third run fired for the bump commit itself (GitHub Actions doesn't re-trigger on pushes made with the built-in `GITHUB_TOKEN`, which reinforces the `startsWith` guard).
Generalizes to: any `contains()`-based self-trigger/self-skip guard in CI is fragile the moment a human (or an agent) writes a commit message that discusses the guard itself -- prefer an anchored match (`startsWith`, or a dedicated marker only the bot ever emits) over a bare substring check. Also: a workflow's correctness is only confirmed by watching a real triggered run go green, never by parsing the YAML alone -- this bug was invisible in the YAML and only visible in `gh run list` output.
